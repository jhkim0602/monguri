# 멘토 페이지 전면 리팩토링 구현 계획

## 1. 방향
- **모듈화**: mentor 도메인 로직/컴포넌트를 재사용 가능한 구조로 분리
- **데이터 정합성**: 멘토 액션이 멘티 UI에 즉시 반영되는 단일 소스 구축
- **점진적 전환**: 기존 멘티 화면 유지 + mentor 영역만 먼저 교체

## 2. 디렉토리 구조(제안)
- `src/app/(mentor)/mentor/...` : 라우팅 레이어
- `src/features/mentor/` : 도메인 컴포넌트/훅/서비스
- `src/features/shared/` : 멘토/멘티 공용 컴포넌트
- `src/lib/api/` : API 클라이언트, fetch 래퍼
- `src/types/` : 공용 타입/스키마

## 3. 상태/데이터 계층
- API 기반으로 전환(REST 계약 우선)
- 클라이언트 캐시/동기화는 **간단한 fetch + SWR/React Query** 중 택1
- 멘티 화면: 기존 mock/로컬스토리지 → API 동기화 레이어 추가

## 4. 리팩토링 전략
1) **멘토 페이지 제거/교체**
   - 기존 `src/app/(mentor)/mentor/*` 페이지는 새 구조로 대체
2) **멘티 연동 포인트 확보**
   - 멘티 Home/Planner/Calendar/Columns에 API 데이터 연결
3) **Mock → API 전환**
   - mock 데이터는 fixtures/seed로 이동

## 4-1. 멘토 플로우 우선순위
1) 과제 → 2) 플래너 → 3) 칼럼

## 5. 기능별 구현 단계
### Phase 0: 인프라
- 인증/권한 가드(멘토/멘티 분리)
- 공용 API 클라이언트, 에러 핸들링

### Phase 1: 과제/피드백
- 과제 CRUD, 배정, 상태 전환
- 제출물/피드백 작성 플로우

### Phase 2: 캘린더/플래너 모아보기
- 학생 캘린더 조회
- 플래너 모아보기 카드/상세
- 플래너 종합 피드백 작성 및 멘티 반영

### Phase 3: 칼럼/글작성
- 칼럼 시리즈/작성/발행
- 멘티 칼럼 리스트 연동

### Phase 4: 대시보드/학생 상세
- 통계 카드, 활동 로그
- 학생 상세 탭(프로필/과제/캘린더/플래너)

## 6. 테스트/검증
- 핵심 플로우 e2e(과제 생성→배정→제출→피드백)
- 캘린더/플래너 조회 및 코멘트 반영
- 칼럼 발행 후 멘티 리스트 반영

## 7. 마이그레이션/데이터 이행
- 로컬스토리지 키(`planner-day-tasks`, `mentee-calendar-events`, `mentor-meeting-records`)는 API 응답에 매핑
- 기존 mock은 fixtures로 보관하고 개발 환경에서만 사용
